2. Methodology

Execution Analysis

Both programs are executed through Refinery's SyntheticRunner, a calibrated emulator that derives instruction counts, memory utilisation, and I/O operation counts from COBOL AST features. Telemetry is deterministic for a given source. In production engagements, Refinery runs against a live Hercules/MVS z/Architecture emulator for hardware-accurate SMF telemetry.

Semantic Analysis

  1. COMPUTE Expression Analysis: All COMPUTE arithmetic statements are extracted and compared. Any change in a COMPUTE expression is flagged at HIGH severity.
  2. Data Type Declarations: PIC clause declarations are compared field by field. Storage-format changes that preserve precision (PIC 9 to COMP-3) are LOW severity. Field size changes are HIGH severity.
  3. Verb Count Analysis: PERFORM, SEARCH, and SORT verb counts are compared. Control-flow divergence is MEDIUM severity.

GnuCOBOL Proxy Limitations

The equivalence check (check 6) runs on x86 hardware under GnuCOBOL, not a native z/Architecture environment. Known divergences from real mainframe execution: (a) EBCDIC/ASCII — GnuCOBOL uses ASCII; character comparisons against EBCDIC literals may produce different results. (b) Packed decimal (COMP-3) — edge cases in packed decimal arithmetic and 0C7 abend conditions differ between GnuCOBOL and z/Architecture instruction execution. (c) Floating-point rounding — GnuCOBOL uses IEEE 754 (x86); z/Architecture uses IBM hex floating point (HFP), which can diverge for COMP-1/COMP-2 fields. (d) I/O simulation — VSAM and DASD I/O is simulated; sequential file behaviour is representative, not bit-identical. Where Hercules/MVS telemetry is available (RealHerculesRunner), these limitations do not apply.

Verdict Criteria

PASS requires all checks clean. Any HIGH or MEDIUM finding blocks deployment. LOW-only findings permit approval.
